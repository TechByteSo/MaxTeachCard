# Definition of Done (DoD)

Универсальный чеклист готовности **статического** сайта: лендинг, визитка, корпоративный сайт **без серверной БД** и без CMS. Подходит для сдачи заказа и внутреннего контроля качества.

---

## 1. Семантика и HTML

- [ ] Используются региональные теги: `<header>`, `<main>`, `<footer>` (и при необходимости `<nav>`, `<section>`).
- [ ] Ровно **один** логичный `<h1>` на страницу.
- [ ] Логичная иерархия заголовков: `<h2>` → `<h3>` без пропусков уровней.
- [ ] Интерактив «как кнопка» — это `<button type="button">` или `<a href="...">` с корректными атрибутами (не «голый» `div` с `onclick` без крайней необходимости).
- [ ] У полей форм есть `<label>` с связкой `for` / `id`.
- [ ] Все заявленные блоки/страницы присутствуют; нет пустых заглушек без договорённости с заказчиком.
- [ ] Тексты вычитаны: опечатки, единый стиль, актуальные даты и контакты.
- [ ] Изображения с осмысленным `alt`; декоративные — с пустым `alt=""` или `aria-hidden`, если уместно.
- [ ] Ссылки на внешние PDF/файлы открываются корректно; пути относительные или абсолютные согласованы с хостингом.
- [ ] **Контактные данные кликабельны** (телефон: `tel:`, email: `mailto:`).

---

## 2. Вёрстка и адаптив

- [ ] Проверка на типичных ширинах: **320px**, **768px**, **1024px**, **1280px** (и при необходимости промежуточные).
- [ ] Нет горизонтального скролла из-за вёрстки (кроме осознанных элементов).
- [ ] Текст и интерактив **не выходят** за границы карточек и контейнеров.
- [ ] Изображения без нежелательного растягивания/сплющивания (`object-fit` / пропорции по ТЗ).
- [ ] Мобильное меню открывается, закрывается и не ломает страницу.
- [ ] Бургер-меню на мобильных можно закрыть не только повторным нажатием на иконку, но и **кликом по свободной области вне меню** (оверлей, фон страницы), без залипания состояния; при закрытии согласованы `aria-expanded` и визуальное состояние кнопки.
- [ ] Кликабельные зоны не меньше разумного минимума (~44×44 px там, где критично для тача).
- [ ] Фиксированный header не перекрывает якорные секции (`scroll-margin` / отступ сверху).

---

## 3. Дебаг, консоль и сеть

- [ ] В **Console** нет необработанных красных ошибок при типичном сценарии использования.
- [ ] Во вкладке **Network** нет неожиданных **404** на ресурсах страницы (стили, скрипты, шрифты, картинки, иконки).
- [ ] Все картинки и иконки (в т.ч. из CSS и шрифтов) реально подгружаются на проде.

---

## 4. Поведение и интерактив (JS)

- [ ] Меню, якоря, слайдеры, модальные окна, раскрывающиеся блоки — работают согласно ТЗ.
- [ ] Слайдеры: автопереключение только если это оговорено (иначе — только ручное).
- [ ] Формы: валидация на клиенте (где нужно), понятные сообщения; отправка — по ТЗ (заглушка / сервис / относительный endpoint).
- [ ] **Если форма реальная – тестовое письмо отправляется** (проверено заказчиком или вами).
- [ ] Состояния UI согласованы с доступностью: `aria-expanded`, `aria-controls`, `aria-label` для иконок-кнопок.

---

## 5. Доступность (базовый уровень)

- [ ] Видимый фокус с клавиатуры на ссылках и кнопках.
- [ ] Скрытый контент (закрытые модалки, скрытые панели) не остаётся в таб-ордере и не виден «снаружи».
- [ ] Контраст текста и фона достаточен для основного контента (ориентир — WCAG 2.1 AA для типовых блоков).

---

## 6. SEO и соцсети

- [ ] Заполнены уникальные `<title>` и `meta name="description"` на каждой значимой странице.
- [ ] Есть один смысловой `<h1>` (согласован с пунктом про семантику).
- [ ] Ключевые изображения имеют содержательный `alt`.
- [ ] Open Graph: как минимум `og:title`, `og:description`, `og:type`, `og:locale`; при публикации в сеть — `og:url` и `og:image` (если нужен превью-шаринг).
- [ ] Favicon подключён (`.ico` / `.svg` / apple-touch по необходимости).
- [ ] При многостраничности — нет дублей title/description; предсказуемые имена файлов или ЧПУ.

---

## 7. Производительность

- [ ] Изображения сжаты под задачу; ориентир для типичных карточек/блоков — **порядка 300–500 КБ** на файл, hero/LCP — осознанно тяжелее, но без «мегабайтных» файлов без причины (WebP/AVIF где уместно).
- [ ] Подключены только **нужные** начертания шрифтов; нет лишних семейств и весов.
- [ ] Анимации не вызывают заметных лагов на мобильном (проверка на реальном устройстве или throttling в DevTools).
- [ ] Крупные изображения вне первого экрана — `loading="lazy"` где не LCP.
- [ ] Нет лишних CSS/JS; для критичного шрифта — разумный `font-display`.
- [ ] **Проверка через PageSpeed Insights – хотя бы жёлтый (50+).**

---

## 8. Безопасность и приватность (статика)

- [ ] В репозитории и в коде **нет** паролей, API-ключей, токенов.
- [ ] Внешние скрипты — из доверенных источников; версии зафиксированы или осознанно выбраны.
- [ ] Формы с персональными данными — по ТЗ: куда уходят данные, есть ли согласие/политика при необходимости.

---

## 9. Сборка, деплой и сдача

- [ ] Сайт выложен на **GitHub Pages**, другой статический хостинг или передан **архив** — по договору.
- [ ] Сайт открывается с **корректного базового URL** (корень или подпапка); пути работают на проде (регистр файлов на Linux).
- [ ] **HTTPS**; нет смешанного контента (mixed content).
- [ ] Страница **404** или редирект — по договорённости; при необходимости `robots.txt` и `sitemap.xml` для многостраничных сайтов.
- [ ] Заказчику отправлено сообщение с результатами: **как открыть сайт**, **как проверить адаптив** и **как проверить форму** (если есть).
- [ ] **Все ссылки открываются в новой вкладке, если это указано** (атрибуты `target="_blank"` с `rel="noopener noreferrer"`).

### 9.1. Pre-host для REG.RU (практический минимум)

- [ ] В `index.html` заполнены `og:url` и `og:image` **реальным доменом** (не `your-domain.ru`).
- [ ] В корне есть и настроены: `.htaccess`, `robots.txt`, `404.html`, favicon (`.svg`/`.ico`).
- [ ] В `.htaccess` нет дублей правил, включены gzip и базовое кеширование статики.
- [ ] Нет очевидных битых путей к локальным ресурсам (изображения, PDF, CSS, JS, favicon).
- [ ] Для `target="_blank"` указан `rel="noopener noreferrer"`.
- [ ] Форма либо подключена к рабочему endpoint/сервису, либо явно помечена как заглушка до подключения backend.
- [ ] Перед загрузкой в REG.RU проверен регистр имён файлов/папок (Linux case-sensitive), особенно в `images/` и `documents/`.

---

## 10. Качество кода и сопровождение

- [ ] Нет мёртвого кода и закомментированных «полуфабрикатов» без пометки.
- [ ] Единый стиль; понятные имена классов и файлов.
- [ ] Краткий **README**: как открыть локально, куда заливать, что менять при смене домена (опционально).
- [ ] (Опционально) минифицированные `.min.css` / `.min.js` и исходники для правок.

### 10.1. БЭМ и согласованность классов (опционально)

**БЭМ (Блок, Элемент, Модификатор)** — методология именования классов, которая делает код предсказуемым и масштабируемым.

#### Соглашения

- **Блок** — независимый компонент: `.card`, `.header`, `.menu`.
- **Элемент** — часть блока, не используется отдельно: `.card__title`, `.card__image`, `.menu__item`.
- **Модификатор** — флаг, меняющий внешний вид или состояние: `.card--featured`, `.menu__item--active`, `.button--disabled`.

#### Чек-лист

- [ ] Имена классов следуют БЭМ-синтаксису: `block`, `block__element`, `block--modifier`, `block__element--modifier`.
- [ ] Блоки имеют уникальные имена (нет конфликта с другими блоками).
- [ ] Один и тот же фрагмент UI в **HTML, CSS и JS** использует **согласованные** имена классов.
- [ ] Модификаторы состояния (открыто, активно, скрыто) именуются предсказуемо: `menu--open`, `accordion__item--active`, `modal--visible`.
- [ ] Глобальные классы типа `.active`, `.hidden` без привязки к блоку **не используются** (допустимы только в utils, если это оговорено).

#### Почему это важно

БЭМ снижает риск случайных конфликтов стилей, упрощает поддержку проекта и делает код понятным для другого разработчика.

---

## 11. Приёмка с заказчиком

- [ ] Пройден чеклист на **реальном устройстве** и минимум в **двух браузерах** (например, Chromium + Safari или Firefox).
- [ ] Заказчик подтвердил контент, контакты и юридически значимые формулировки.
- [ ] Переданы файлы / доступ к репозиторию / хостинг — по договору.

---

## Краткая формула готовности

Сайт готов, если он **семантичен и доступен на базовом уровне**, **корректен на ключевых ширинах**, **без ошибок в консоли и без 404 ресурсов**, **с базовым SEO и favicon**, **с разумной перфоманс‑дисциплиной**, **выложен или передан заказчику с понятной инструкцией проверки**, а контент и юридические моменты **согласованы**.
